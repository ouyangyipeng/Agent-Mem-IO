## 一、赛题内容详细解释

### 1.1 赛题基本信息

- **赛事名称**: 2026年全国大学生计算机系统能力大赛-操作系统设计赛(全国)-OS功能挑战赛道
- **赛题名称**: 面向Agent记忆的向量检索系统I/O优化
- **赛题难度**: B级（学术型）
- **主办单位**: 南开大学

### 1.2 赛题背景

大模型Agent的"长时记忆"面临两大挑战：

1. **规模庞大**: 记忆数据远超物理内存容量
2. **高度动态**: Agent不断产生新记忆（写入）并回忆历史（读取）

当高维向量被组织成图索引（如HNSW）存储于SSD时，传统OS文件I/O机制失效：
- **随机读问题**: 图遍历检索导致极度随机读，OS的顺序预取和Page Cache完全失效
- **写放大问题**: 实时高频随机写导致严重的写放大

### 1.3 核心任务

设计并实现一个针对动态向量检索的底层I/O存储引擎，需要完成：

| 任务类型 | 具体要求 |
|---------|---------|
| **读优化-缓存设计** | 放弃OS Page Cache，在用户态实现高效缓存池，设计合理的缓存淘汰策略 |
| **读优化-I/O预取** | 结合图索引拓扑连通性，设计"下一跳"预取逻辑，使用io_uring异步框架 |
| **写优化** | 借鉴LSM-Tree思想，将随机写转化为顺序追加写，实现后台异步合并 |

### 1.4 核心约束

| 约束项 | 要求 | 说明 |
|--------|------|------|
| **内存限制** | 数据集大小的10%-20% | 如SIFT1M（~500MB）需限制内存在50-100MB |
| **召回率** | Recall@10 ≥ 85% | 硬性指标，低于此值系统不合格 |
| **禁止mmap** | 严禁使用mmap | mmap依赖Page Cache，与赛题要求背道而驰 |
| **必须O_DIRECT** | 必须使用O_DIRECT打开文件 | 绕过OS缓存，自己接管内存和磁盘调度 |

## 二、赛题意义

### 2.1 学术意义

本赛题涉及多个前沿研究方向：
- **向量检索**: 近似最近邻搜索（ANN）是机器学习领域的核心问题
- **I/O优化**: 突破传统OS存储栈的限制，实现用户态I/O调度
- **图索引**: HNSW/DiskANN等图索引算法是当前最优的ANN方案

### 2.2 工程意义

- **Agent应用**: 为大模型Agent提供高效的长期记忆存储
- **数据库系统**: 探索向量数据库的底层I/O优化方案
- **操作系统**: 挑战传统OS I/O栈的局限性

## 四、评分方式详解

### 4.1 评分模块

根据赛题文档，评分分为四个模块：

```
总评分 = 性能指标(25%) + 创新性(25%) + 代码质量(25%) + 文档完整性(25%)
```

#### 4.1.1 性能指标 (25%)

核心考核：
- **吞吐量(QPS)**: 高并发混合读写下的查询每秒数
- **延迟**: P99和P99.9延迟，关注毛刺
- **召回率(Recall@10)**: 查询准确性，必须≥85%

#### 4.1.2 创新性 (25%)

评估I/O优化的新颖程度：
- 是否有独特的缓存淘汰策略
- 是否有创新的预取算法
- 是否有高效的写优化方案

#### 4.1.3 代码质量 (25%)

评估标准：
- 代码规范性
- 测试覆盖率
- 内存安全（无泄漏）

#### 4.1.4 文档完整性 (25%)

要求文档：
- 架构设计文档
- 性能测试报告
- 学术论文

### 4.2 初赛与决赛

- **初赛**: 按照公开测试用例进行评测
- **决赛**: 追加新的测试用例，可能包括：
  - 更大规模数据集
  - 更严格的内存限制
  - 更高的并发要求

## 七、本地测试指南

### 7.1 环境要求

- **操作系统**: Linux (推荐Ubuntu 22.04)
- **编译器**: GCC 11+ 或 Clang 14+
- **CMake**: 3.16+
- **内存**: 建议16GB+
- **存储**: NVMe SSD（用于测试I/O性能）

### 7.2 构建步骤

```bash
# 1. 克隆项目
git clone https://github.com/ouyangyipeng/Agent-Mem-IO.git
cd Agent-Mem-IO

# 2. 创建构建目录
mkdir build && cd build

# 3. 配置CMake
cmake ..

# 4. 编译
make -j$(nproc)

# 5. 运行测试
./agent_mem_io_tests

# 6. 运行基准测试
./agent_mem_io_benchmark -n 10000 -d 128 -q 100 -k 10
```

### 7.3 清除缓存测试

为了模拟真实评测环境，每次测试前应清除OS缓存：

```bash
# 清除OS缓存（需要root权限）
sync && echo 3 | sudo tee /proc/sys/vm/drop_caches

# 然后运行测试
./agent_mem_io_benchmark -n 10000 -d 128 -q 100 -k 10
```

### 7.4 内存限制测试

使用cgroups或ulimit限制内存：

```bash
# 使用ulimit限制内存为100MB
ulimit -v 102400
./agent_mem_io_benchmark -n 10000 -d 128 -q 100 -k 10
```

## 九、技术方案详解

### 9.1 已实现的技术

#### 9.1.1 NSW图索引

```
原理: 可导航小世界图
优势: 
  - 对数级搜索复杂度 O(log N)
  - 支持增量构建
  - 适合内存受限场景
```

#### 9.1.2 O_DIRECT + io_uring

```
O_DIRECT: 绕过OS Page Cache
io_uring: Linux 5.1+异步I/O框架
优势:
  - 减少内存拷贝
  - CPU与I/O并行
  - 批量I/O提交
```

#### 9.1.3 LSM-Tree写入

```
组件:
  - MemTable: 内存写入缓冲
  - SSTable: 磁盘有序文件
  - Compaction: 后台合并
优势:
  - 顺序写代替随机写
  - 写放大降低
```

### 9.2 待优化的技术

| 技术方向 | 说明 | 预期收益 |
|---------|------|---------|
| **量化压缩(PQ/SQ)** | 压缩向量存储 | 内存降低4-8倍 |
| **分层NSW(HNSW)** | 多层图结构 | 搜索速度提升 |
| **图剪枝** | 减少边数 | 内存降低 |
| **智能预取** | 基于访问模式 | 延迟降低 |

## 附录：常见问题

### Q1: 为什么不用mmap？

mmap依赖内核的缺页中断和Page Cache，在向量检索的随机访问模式下会产生大量Page Fault，性能极差。赛题明确要求"放弃面对向量检索低效的操作系统默认机制"。

### Q2: io_uring和传统aio有什么区别？

- **io_uring**: Linux 5.1+原生支持，零拷贝、完全异步、支持批量提交
- **aio**: 传统POSIX异步I/O，有限制，不支持缓冲I/O

### Q3: 为什么Recall@10要≥85%？

这是赛题的硬性指标。召回率过低意味着检索结果不准确，系统失去实用价值。

### Q4: 如何验证内存限制？

使用`/proc/<pid>/status`查看VmRSS，或使用valgrind检测内存泄漏。
