## 1. 赛题核心挑战分析

### 1.1 问题本质

大模型 Agent 的长时记忆存储面临三大核心挑战：

```
┌─────────────────────────────────────────────────────────────────┐
│                    Agent Memory I/O 挑战                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐      │
│  │  极度随机读   │    │  高频实时写   │    │  内存受限    │      │
│  │  (图遍历)     │    │  (记忆更新)   │    │  (10%-20%)   │      │
│  └──────┬───────┘    └──────┬───────┘    └──────┬───────┘      │
│         │                   │                   │              │
│         ▼                   ▼                   ▼              │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │              传统 OS I/O 机制完全失效                     │  │
│  │  • Page Cache 顺序预取无效 → Page Fault 风暴             │  │
│  │  • 随机写导致写放大 → SSD 寿命下降                       │  │
│  │  • mmap 缺页中断开销巨大 → 延迟不可控                    │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 为什么传统方案失效？

| 传统方案 | 失效原因 | 本项目解决方案 |
|----------|----------|----------------|
| `mmap` + Page Cache | 图遍历的随机访问模式导致大量缺页中断，预取完全无效 | `O_DIRECT` + 用户态 Buffer Pool |
| `pread/read` | 同步阻塞 I/O，CPU 等待磁盘延迟 | `io_uring` 异步非阻塞 |
| 顺序预取 | 图索引的"下一跳"是跳跃式访问，非顺序 | 拓扑感知预取 |
| 原地更新写 | 随机写导致 SSD 写放大 | LSM-Tree 追加写 + 后台 Compaction |

### 2.2 读路径优化 (Read Path)

```
查询流程 (以 Top-K 搜索为例):

1. 接收 Query 向量
2. 从入口点开始图遍历
3. 对于当前访问节点:
   ┌─────────────────────────────────────────────────────────────┐
   │  CPU 计算: 当前节点与 Query 的距离                           │
   │     ↓ (并行)                                                │
   │  I/O 预取: 通过 io_uring 异步拉取邻居节点的向量数据          │
   │     ↓                                                       │
   │  当 CPU 完成计算时，邻居数据已在 Buffer Pool 中              │
   └─────────────────────────────────────────────────────────────┘
4. 选择下一跳节点，重复步骤3
5. 返回 Top-K 结果
```

**关键创新点**:
- **拓扑感知预取**: 利用图的邻接关系，提前知道下一跳可能访问的节点
- **计算-I/O 重叠**: CPU 计算距离的同时，I/O 引擎在后台拉取数据
- **Buffer Pool 热点缓存**: 基于图访问频次的智能缓存淘汰

### 2.3 写路径优化 (Write Path)

```
写入流程 (LSM-Tree 风格):

1. 新向量写入 → MemTable (内存中的有序结构)
2. MemTable 满了 → 转为 Immutable MemTable
3. 后台线程将 Immutable MemTable 刷写到 SSD (顺序写)
4. 后台 Compaction 合并多个 SSTable，重建图索引

关键设计:
- 写入是顺序追加，避免随机写
- Compaction 在后台异步进行，不阻塞查询
- 图索引重建时保持连通性
```

## 3. 关键技术决策

### 3.1 为什么选择 Vamana 图索引？

| 特性 | HNSW | Vamana (DiskANN) | 本项目选择 |
|------|------|------------------|------------|
| 内存占用 | 高 (全量图在内存) | 低 (仅导航结构在内存) | ✅ Vamana |
| 随机读模式 | 极度随机 | 可预测的邻居访问 | ✅ Vamana |
| SSD 友好度 | 一般 | 专门优化 | ✅ Vamana |
| 召回率 | 高 | 高 | 相当 |

### 3.2 为什么选择 io_uring？

```
传统 I/O 模型对比:

┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│  同步阻塞 I/O    │     │  libaio         │     │  io_uring       │
│  (pread/pwrite) │     │  (Linux AIO)    │     │  (Linux 5.1+)   │
├─────────────────┤     ├─────────────────┤     ├─────────────────┤
│  • 系统调用开销大│     │  • 仅支持 O_DIRECT│     │  • 零系统调用   │
│  • CPU 空闲等待  │     │  • API 复杂     │     │  • 批量提交     │
│  • 无法流水线   │     │  • 不支持缓冲读  │     │  • 完成通知高效 │
│  ❌ 不适合       │     │  ⚠️ 可用但非最优 │     │  ✅ 最佳选择    │
└─────────────────┘     └─────────────────┘     └─────────────────┘
```

### 3.3 Buffer Pool 淘汰策略选择

**候选方案分析**:

1. **LRU (Least Recently Used)**
   - 优点: 实现简单
   - 缺点: 对图遍历的"扫描"模式不友好，可能淘汰热点

2. **2Q (Two Queues)**
   - 优点: 区分"扫描"和"热点"访问模式
   - 缺点: 需要调参 (两个队列的比例)

3. **Graph-Aware LRU (本项目创新)**
   - 思路: 结合图的拓扑结构，优先保留"枢纽节点" (高入度节点)
   - 实现: 维护节点的访问频次 + 入度信息
   - 预期效果: 更高的缓存命中率

**决策**: 先实现 2Q 作为基线，后续迭代优化为 Graph-Aware 策略。

### V6.1综合性能优化：io_uring线程安全 + PQ ADC阈值 + entry point pinning

**日期**: 2026-04-23

**问题**: V5/V6审计发现3个问题：
1. V5-Minor-1: README SSD QPS数据需更新（6.12→实测7.62+）
2. V5-Minor-2: 4线程缓存命中率0.2%（vs 单线程66.8%），多线程竞争小缓存
3. V6-NEW-1: 多线程SSD搜索禁用io_uring（async_buffers_线程不安全）

**设计决策**：
- io_uring async_buffers_添加mutex保护（安全修复），但多线程仍传nullptr给io_engine
  - 原因：共享io_uring ring导致completion stealing，实测4线程QPS从9.25→0.90
  - mutex是安全措施（防止数据竞争），但共享ring性能是灾难性的
  - 多线程I/O并行性来自并发同步读取（BufferPoolManager线程安全）
- PQ ADC阈值截断：当results.size()>=k时，截断PQ距离>worst_dist×4.0的候选
  - 减少不必要的SSD读取，单线程QPS从6.48→7.62（+17%）
- Entry point pinning：搜索前pin entry point，搜索后unpin
  - 保护最关键节点不被淘汰，仅1页不会导致cache lock
- beam_width保持8：实测beam_width=16对10K数据集QPS更低（5.10 vs 6.48）

**修改文件**：
1. `src/io/disk_layout.h` — 添加`std::mutex async_buffers_mutex_`保护async_buffers_
2. `src/io/disk_layout.cpp` — submit_async_batch()加锁、wait_async_batch()细粒度加锁（避免死锁）
3. `src/benchmark.cpp` — PQ ADC阈值截断、entry point pinning、多线程注释更新
4. `README.md` — SSD QPS更新为实测值（7.62/9.25/5.38）
5. `PROGRESS.md` — QPS追踪表更新
6. `docs/REPORT.md` — 性能数据更新
7. `docs/PAPER.md` — 性能对比表更新

**实测结果**：
- SIFT 10K 1线程: QPS=7.62 (↑24% vs 6.12), Recall=98.90%, Cache=66.7%
- SIFT 10K 4线程: QPS=9.25 (↓8% vs 10.10, 但更诚实), Recall=98.90%, Cache=0.3%
- SIFT 100K 1线程: QPS=5.38 (↑11% vs 4.85), Recall=99.60%, Cache=66.9%
- 所有测试PASSED，Memory=20.0% PASS

**教训**：
- 共享io_uring ring在多线程下性能灾难（completion stealing），mutex只解决安全不解决性能
- beam_width不是越大越好，小数据集beam_width=8更优（减少SSD读取量）
- PQ ADC阈值截断是真实有效的QPS优化（减少不必要的SSD读取）
